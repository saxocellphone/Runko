# the status quo

You've already built the workarounds

Monorepos are winning and coding agents are multiplying. The tooling hasn't caught up, so teams improvise:

Every project is a snowflake

Each new service starts as a copy of the last one. Two years in, no two projects share a layout, an owners file, or a CI setup — and there's no way to fix them all at once.

Agents live in git worktrees

One worktree per agent works — until you're babysitting five branches, half-finished work is stranded on one machine, and every agent pushes with your credentials.

Merge policy is theater

CODEOWNERS suggests. Branch protection half-enforces. Stacked PRs are faked with base branches, and a one-line change re-runs the whole CI matrix.

$ runko project create

Projects come from a generator, not a copy-paste

Name, type, owners in — everything else is generated or defaulted. That's the whole form.

Uniformity isn't a style guide anyone has to follow — it's the generator everyone uses. Every project in the org has the same shape: a manifest next to the code, a skeleton from your org's template, a build binding with zero hand-written lines.

Ownership and required checks live in PROJECT.yaml, in the tree — so changing the rules is itself a reviewed change that owners approve, and the whole platform re-reads policy from the code it protects.

$ runko project create payments-api \
    --type service --owners team-payments
  create payments-api/PROJECT.yaml  # type + owners
  create payments-api/main.go       # org skeleton
  create payments-api/BUILD.bazel   # generated
$ runko change push
  # the new project lands like any other change

$ runko --json

The whole platform is a CLI

Web UI optional. Every flow — projects, workspaces, review, landing — is a command that speaks JSON and tells you what to do next when it refuses.

$ runko workspace create --name fix-sku \
    --project payments-api
  materialized payments-api/  # ms, not minutes
$ $EDITOR payments-api/sku.go
$ runko change push
  I4f2a…e1 → review  # checks scoped to the change
$ runko change requirements --json
  { "checks": {"pending": []}, "mergeable": true }
$ runko change land
  # rebase-lands behind the gates

Refusals are answers. Every failure is structured — {code, message, suggestion} — and the suggestion is a command you can type. Exit codes mean something. --json is everywhere, so piping to jq replaces screen-scraping.

One interface, three audiences. Engineers, coding agents, and CI use the same two binaries: runko for people and agents (a generated AGENTS.md teaches it), runko-ci for pipelines — affected computes what a change touches offline, checkout does partial-clone sparse checkouts, report-check posts results back with retries.

Prebuilt binaries for Linux, macOS, and Windows ship on every CLI-affecting landing; the CLI contract is a versioned document, not folklore.

$ runko agent create

Run five agents like it's one

Everyone found the same trick: a git worktree per agent. Runko workspaces are what that trick wants to be — each agent gets its own slice of the repo, its own identity, and its own guardrails.

Git worktrees todayRunko workspaces
CheckoutA full tree per worktreeOne shared clone; each workspace materializes only its slice
Work in progressStranded on one machineSnapshotted to the server — kill the box, re-attach anywhere
IdentityEvery agent pushes as youOne command mints a per-task identity — named for the work, its own token, dead by TTL
GuardrailsWhatever your token allowsPath allowlists, per-change size caps, secret scan, no self-approval — enforced at the server
LifecycleStale worktrees pile upOne task, one workspace, enforced — auto-closed when its last change lands or is abandoned
Getting code inHand-merge the branches laterEach agent's work arrives as stacked, reviewable changes behind the same gates as humans — oversized changes are refused with the split workflow in the message

The agent's whole interface is the same CLI your engineers use — taught by a generated AGENTS.md, with policy enforced where it can't be prompt-engineered away:

$ runko agent create --task fix-sku
  # ephemeral identity: agent-fix-sku-3f2a, dies by TTL
$ runko change push
  # stack's tip → review; the whole stack updates
$ runko change requirements --json
  # owners + checks outstanding, as structured JSON
$ runko change automerge
  # arm once — the server lands it when gates go green

$ runko change land

And landing works like it should

Everything below is shipped and tested against a real daemon, real Git, and a real Postgres — not a roadmap.

Stacked changes, jj-native

Jujutsu is the primary client: rework the root of a stack, descendants rebase themselves, and one push updates every change on the server. Plain git push stays fully supported.

Gates that can't be skipped

Owners and required checks are declared next to the code they protect. Approvals bind to the exact commit reviewed — amend it, and approval is re-earned, not carried over.

Land without re-running the world

Rebase-based landing re-runs checks only when trunk actually touched something your change depends on. A lost race is a retry, never a silent overwrite.

CI runs only what can break

Changed paths map to projects and their dependents, fail-closed, with optional build-graph refinement (Bazel first). Your runners, your pipelines — we never run your CI.

One door into trunk

Direct push is closed. Every write — human, agent, or bot — passes the same policy, secret scanning (gitleaks), and change tracking on the server.

Arm it and walk away

Automerge is a bit, not a babysitter: arm a change once and the server lands it the moment owners and checks go green — through the same gates, attributed to whoever armed it, never forced.

Releases are immutable

runko release create versions a project, derives the changelog from the changes that actually landed, and writes the tag server-side under tag policy. No edit, no delete — a wrong release is followed by a corrected one.

Checks declare their class

run_when: direct lanes run only when a project's own code changes; integration lanes still ride the dependency closure. A dependent's unit tests stop re-running for changes that can't break them.

Mirror to any git host

Adopt gradually: keep a leased, divergence-guarded mirror on GitHub, GitLab, Gitea — anything that speaks git. Trust lives where you want it; truth stays yours.

$ docker compose up

Self-hosted, honest about scope

  • Git is the only substrate. No custom object store, no virtual filesystem — your escape hatch is git clone, forever.
  • We never run your CI. Runko owns change identity, webhooks, the Checks API, and affected computation; your runners and pipelines stay yours.
  • Postgres is a rebuildable index. Burn it down and re-derive everything from the tree.
  • Apache-2.0. One binary + Postgres via docker compose up; Kubernetes if you feel fancy.

Getting started

Your org already runs Runko; you bring one CLI. From joining to a landed change in five steps — the last one repeats forever, and the journey is replayed end-to-end in CI, refusals included, so it can't rot.

  1. Kick the tires without installing anything

    The playground runs the full UI on sample data in your browser, and this instance's own source is browsable with no account. Want in here? Request an invite.

  2. Install the CLI

    Prebuilt binaries for Linux, macOS, and Windows, refreshed on every CLI-affecting landing — or go install it. Your org's control plane is already up; ask your operator for its URL.

  3. Sign up — signup is login

    One command against your org's control plane creates your account and stores the credential — no browser round-trip, no token pasting. Already have an account? runko auth login.

  4. Land on day one

    Make a workspace — your slice of the monorepo, not a second clone — generate a project, push the change it opens, land it when the gates go green. When something refuses, the refusal names the command that fixes it.

  5. Read the rules like code

    Owners, required checks, and agent policy all live in the org's tree — browsable like any other file, and changing policy is itself a reviewed change. (Running the control plane yourself is docker compose up — the operator's path, not day one's.)

$ runko auth signup --runkod-url https://runko.acme.dev \
    --name val --org acme --join
  # account created, login stored; signup is login
$ runko workspace create --name day-one --project repo
  materialized repo/  # your slice, ms not minutes
$ runko project create --name hello \
    --type service --api rest
  # scaffold generated, opened as change I5b3c…d7
$ runko change push
$ runko change land
  # landed — day one, no copy-paste, nothing skipped

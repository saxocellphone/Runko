One monorepo that feels small

First-class Projects with owners in the tree, and CitC-class workspaces: a full-repo view that materializes only your slice — sparse cones, partial clone, durable snapshot refs.

Changes that land with confidence

Change-centric review with stable identities that survive rebases. Owners gate by touched paths, required checks come from the tree, and rebase-landing skips re-runs only when it's provably safe.

Humans & agents, co-equal

Every flow has a CLI, an API, and a UI. Agents get generated AGENTS.md, MCP tools, and server-side policy — path allowlists, diff caps, no self-approval — enforced at receive, never on trust.

Built around the workflows that actually hurt

Everything below is shipped and tested against a real daemon, real Git, and a real Postgres — not a roadmap.

⌥Stacked changes, jj-native

Jujutsu is the primary client: rework the root of a stack, descendants auto-rebase, and one push updates every change on the server. Plain git push stays fully supported.

✓Merge gates from the tree

PROJECT.yaml declares owners and required checks; the control plane is a rebuildable index, never a second source of truth. Approvals bind to the exact commit they reviewed.

⚡Optimistic landing

Rebase-based landing with revalidation only when trunk's delta intersects your change's affected set. Race-safe by compare-and-swap — a lost race is a retry, never a silent overwrite.

◎Affected computation

Paths → projects → declared dependents, fail-closed, with optional build-graph refinement (Bazel first). Your CI runs exactly what a change can break — on your runners, not ours.

🛡One receive funnel

Trunk is closed to direct push. Every write — human, agent, or bot — passes policy, secret scanning (gitleaks), and change tracking through the same server-side gate.

⇄Mirror to any git host

Keep a leased, divergence-guarded mirror on GitHub, GitLab, Gitea — anything that speaks git. Trust lives where you want it; truth stays yours.

Agent-native, not agent-tolerant

Coding agents are normal API clients with stricter defaults — enforced where it can't be faked.

# the same three commands your agents learn from the generated AGENTS.md
$ runko change push # stack's tip → review; every change in the stack updates
$ runko change requirements # owners + checks outstanding, as structured JSON with --json
$ runko change land # gates enforced server-side; agents can never approve themselves

# agent guardrails are receive-time policy, not prompt engineering:
agent policy → workspace affinity required · path allow/deny · diff caps · no self-approval · no solo ownership

Self-hosted, honest about scope

  • Git is the only substrate. No custom object store, no virtual filesystem — your escape hatch is git clone, forever.
  • We never run your CI. Runko owns change identity, webhooks, the Checks API, and affected computation; your runners and pipelines stay yours.
  • Postgres is a rebuildable index. Burn it down and re-derive everything from the tree.
  • Apache-2.0. One binary + Postgres via docker compose up; Kubernetes if you feel fancy.

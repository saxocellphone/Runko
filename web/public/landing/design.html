How Runko is designed

Four pillars. Every feature follows from one of these.

01One monorepo that feels small.

All your code lives in one Git repository, but nobody has to carry all of it. You check out just your slice — the rest stays on the server until you need it. Projects and ownership are small files in the tree, so the repo describes itself.

02Trunk-based development, enforced.

There is one branch that matters, and nobody pushes to it directly — not even admins. Every change goes through the same gate: owner approval, required checks, a secret scan. If trunk moved while you waited, Runko only re-runs the checks your change could actually have broken.

03Agents are first-class users.

AI agents use the same CLI and APIs people do — but the server holds them to tighter rules: no approving their own work, no touching paths outside their assignment, hard caps on change size. Enforced server-side, never by trusting the agent.

04Built for jj, works with git.

Jujutsu is the primary client. Every change has a stable identity that survives rebasing and amending, so you can rework the bottom of a stack and everything above it follows — one push updates the whole stack. Plain git works too; you just do the rebasing yourself.
